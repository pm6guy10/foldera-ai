## 37. When Multiple CSS Variable Systems Compete, `grep var(--prefix-)` Across the Whole Repo Is the Only Reliable "Is It Live" Test

`app/globals.css` had accumulated three full token systems (`--foldera-*`, `--ld-*`, and the `.fd`-scoped `--bg`/`--fg*`/`--cyan` system) from three successive landing-page redesigns (#378, then the #500 import), each commented as "the" design system at the time it was written. Comments and section headers ("Self-contained design layer... binding") were not reliable signals of which one was actually live — `--ld-*` still carried a confident-sounding doctrine comment but had zero usages anywhere in `.tsx`. The only reliable test was `grep -rn "var(--prefix-" --include="*.tsx" --include="*.css" .` for every prefix, then tracing each surviving hit to confirm its consuming component is actually reachable from a route (`components/foldera/RightPanel.tsx` defined CSS-matched classes but was never imported anywhere — dead code referencing a dead variable looks identical to live code at a glance).

**Rule:** when a CSS file shows comments describing multiple "the design system" blocks, do not trust the most recent or most detailed comment — grep every variable prefix's `var()` usage repo-wide and trace each consumer to an actual rendered route before deciding what's live. A variable with zero `var()` references outside its own declaration block is deletable regardless of how authoritative its surrounding comment sounds.

## 38. "Collapse N Raw Values Onto M Tokens" Is Only Safe When the Token and the Literal Resolve to the Identical Value — Fluid Tokens Are Not a Drop-In Replacement for Fixed Ones

Asked to collapse ~37 raw `font-size` px values and ~22 raw `border-radius` values onto a small set of design tokens, the literal-minded move (snap every value to its nearest token) would have silently changed rendered output: the project's `--fs-display/h1/h2/h3/body` tokens are `clamp()`-based fluid values that vary by viewport, not fixed px — replacing a raw `21px` heading with `var(--fs-h3)` doesn't just rename it, it makes it responsive (different actual size at every viewport other than the one where the raw value happened to match). Only `border-radius: 999px/9999px → var(--r-pill)` and `font-size: 13px/11px → var(--fs-caption/mono-label)` were truly zero-risk, because `--r-pill` and the two small `--fs-*` tokens are fixed values that resolve to the exact literal being replaced, in every browser, at every viewport, with no exceptions. Everything else in scope (8/11/12/18/24px radii component-scaled by the imported design; assorted rem/px body sizes; the fluid heading clamps) was left as raw literals rather than guessed onto the nearest fluid token.

**Rule:** before substituting `var(--token)` for a raw literal as a "consolidation," check whether the token's CSS value is a fixed constant or a `clamp()`/`calc()` expression that varies by context. A fixed-to-fixed substitution at an exactly-matching value is a true no-op. A fixed-to-fluid substitution is a behavior change wearing a refactor's clothes — it must be flagged and left out of "consolidation, not redesign" scope rather than silently applied because the numbers were "close enough."
