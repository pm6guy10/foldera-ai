# Foldera — Innovation Audit

**Principal engineer / product architect view.** Not bug-hunting; identifying missed leverage.

---

## 1. Intelligence ceiling

*Where the system makes dumb decisions it could make smarter with data it already has.*

### 1.1 Extracted but never stored: goals and outcomes

**Location:** `lib/extraction/conversation-extractor.ts`

The extractor asks Claude for **goals** and **outcomes** (decision_description + result) and parses them into `payload.goals` and `payload.outcomes`. Only `decisions` and `patterns` are written to the graph. **Goals are never inserted into `tkg_goals`.** **Outcomes are never persisted anywhere.**

- **Impact:** The conviction engine and artifact generator read goals only from `tkg_goals` (onboarding + priorities). Every goal and outcome inferred from conversations and email is discarded. The model is told "you have their complete behavioral history" but a whole dimension (stated goals, confirmed outcomes) is missing.
- **Leverage:** Persist `payload.goals` into `tkg_goals` (with source = conversation/email, optional priority). Persist `payload.outcomes` as either (a) updates to existing commitments (match by decision_description, set resolution) or (b) a dedicated outcomes table / signal type. The next directive run would then see "user said they wanted X" and "last time they did Y, result was Z."

### 1.2 Confidence is LLM guesswork, not evidence-based

**Location:** `lib/briefing/generator.ts` (CONVICTION_SYSTEM prompt)

Confidence (0–100) is generated by the model from instructions like "90+ means you have seen this exact scenario resolve this exact way multiple times." The code does **not** compute confidence from:
- Count of similar past directives and their feedback_weight
- Pattern activation_count or recency
- Commitment risk_score or resolution history

So the system has the data to say "this pattern has been approved 5/5 times" or "this action_type has net weight -2.0" but never turns that into a numeric prior. The model is guessing.

- **Leverage:** Add a **confidence prior** in code: e.g. `prior = f(approval_rate_by_type, net_feedback_weight_for_similar_directive, pattern_activation_count)`. Pass it to the prompt: "Suggested confidence prior: 72 (based on 4/5 approvals for send_message and +3.0 net weight). You may adjust ±15 based on qualitative evidence." Or cap model confidence at the prior + 10 so it can't overstate.

### 1.3 Commitments have promisor/promisee; directives don’t use them

**Location:** `tkg_commitments` (promisor_id, promisee_id), `lib/briefing/generator.ts` (commitment query)

The generator selects `description, category, status, risk_score, risk_factors, made_at, source_context` and never joins to `tkg_entities` to resolve **who** the commitment is to or from. So the model sees "Follow up with the client" but not "Follow up with **Marcus** (entity id X, 12 interactions, cooling)."

- **Impact:** send_message directives and "reach out" reasoning can’t be tied to specific entities or to commitments that name those entities. Relationship-aware prioritization (e.g. "you promised Marcus a deliverable; he’s cooling") is underused.
- **Leverage:** In the generator, for active/at_risk commitments, optionally join to entities and add a line per commitment: "To/From: [entity name] (last_contact, trend)." Or pass commitment lines plus a separate "Commitments by relationship" block from a query that groups by promisee_id.

### 1.4 Signals: rich columns, thin usage

**Location:** `tkg_signals` schema vs. conviction/artifact usage

Signals have (or migration defines) `author`, `recipients`, and in some migrations `embedding`, `extracted_entity_ids`, `extracted_commitment_ids`, etc. The conviction engine and artifact generator use only `type`, `source`, `content`, `occurred_at`. So:

- **author/recipients** are used by the relationship tracker (From/To parsing) but not in the directive prompt (e.g. "Last 5 signals were from/to: …").
- **Embeddings** are never written or read; semantic similarity is unused (see §3).

### 1.5 Outcome (worked / didn’t work) is weight-only, not semantic

**Location:** `lib/briefing/generator.ts` (buildFeedbackSection), `app/api/conviction/outcome/route.ts`

Outcome updates only `feedback_weight` (2.0 / -1.5) and `execution_result.outcome`. The feedback section passed to the model lists "EXECUTED — user approved and ran these" with a numeric weight. It does **not** distinguish "user said It worked" vs "user said Didn’t work." So the model sees +2.0 and +1.0 but not the semantic label.

- **Leverage:** In the feedback query, select `execution_result` (or a stored outcome flag). In `buildFeedbackSection`, add a line for executed items: "CONFIRMED WORKED" vs "CONFIRMED DIDN'T WORK" so the model can learn from the quality of outcomes, not just approval.

## 2. Missed feedback loops

*Where the system acts but never learns whether it worked, or where learning is shallow.*

### 2.1 No automatic outcome from the world

**Locations:** `lib/conviction/execute-action.ts`, `app/api/conviction/outcome/route.ts`, Resend webhook

- **Email sent:** When the user approves a send_message directive, the email is sent and `execution_result` stores send status. There is no link from "email we sent for action X" to "user opened it" or "recipient replied." Resend webhook tracks opens for **waitlist** emails only, not for directive-sent emails.
- **Calendar event created:** Event is created via Google/Outlook API; no check later that the event actually occurred (e.g. sync and see "completed" or "cancelled").
- **Document saved:** Stored as a signal; no signal that the user ever opened or used it.

So the only outcome the system gets is **explicit** ("It worked" / "Didn’t work"). There is no passive loop from real-world results.

- **Leverage:** (1) For email: track message_id per sent directive; on next sync, detect replies in the same thread and optionally mark action as "replied" or nudge for outcome. (2) For calendar: after N days, compare "scheduled" vs "occurred" or "no-show" from calendar sync and suggest outcome. (3) Don’t block on this—explicit outcome is already strong; these are incremental.

### 2.2 Outcome route doesn’t write to tkg_signals

**Location:** `app/api/conviction/outcome/route.ts`

Approve/skip in `execute-action.ts` write to both `tkg_actions` and `tkg_signals` (user_feedback, approval/rejection). The outcome route updates only `tkg_actions`. So "It worked" / "Didn’t work" never becomes a **signal** in the same stream as other behavior. Downstream code that only consumes signals (e.g. future extractors or analytics) won’t see it.

- **Leverage:** On outcome POST, insert a `tkg_signals` row (e.g. source=`user_feedback`, type=`outcome_confirmed`, content=action_id + outcome). Idempotent by content_hash. Then the same signal stream that feeds relationship and pattern logic can optionally include outcome confirmations.

### 2.3 User_feedback signals are written but not used for learning beyond tkg_actions

**Location:** `lib/conviction/execute-action.ts`, `lib/briefing/generator.ts`

Approval/rejection is written to `tkg_signals` with rich content (directive + reason). The **next** directive run uses only `tkg_actions` (feedback_weight, approval stats, skip_reason) to build the feedback and approval-rate sections. The **content** of the feedback signals (e.g. free-text skip reason, or the exact directive text) is not re-read from signals; it’s already on the action row. So the signal write is somewhat redundant for the conviction loop—unless you later want to do semantic search over "what did the user reject?" using embeddings. Right now that’s unused (no embeddings).

### 2.4 Learning loop depth

The conviction loop **does** learn: feedback_weight and approval rates drive suppression and boosting. Skip reasons are passed through and the prompt explains them. Outcome (worked/didnt_work) is reflected in weight. So the loop exists; the gaps are (a) no automatic outcome from the world, (b) outcome not in the signal stream, (c) confidence not grounded in this same data.

## 3. Underbuilt leverage points

*Single changes that would lift accuracy or impact.*

### 3.1 One abstraction away: "directive success predictor"

The system has:
- Historical directives with action_type, directive_text, reason, feedback_weight, skip_reason, execution_result.outcome
- Current context (priorities, goals, commitments, patterns, cooling relationships)

What it doesn’t have is an explicit **similarity** step: "Of the directives that were approved and marked 'worked', which are most similar to the candidate we’re about to generate?" That would require either (a) embedding past directives and the current candidate and comparing, or (b) a simple keyword/pattern match. The DB already has `embedding` on commitments and signals in the migration; the application never populates or queries it. So the "one abstraction" is: **embed past directives (or their key fields), and at generation time score the candidate against "what worked" and "what was skipped."** That single layer would make every directive more aligned with proven patterns.

### 3.2 Data source with outsized impact: thread-level email context for send_message

Today the artifact generator gets "relationship history" (entities + patterns) and "recent signals" (last 10, 30 days). It does **not** get "last 3 emails in the thread with Marcus" or "the exact email you’re replying to." So draft quality for send_message is generic by relationship, not by thread. Adding a **thread context** (for a chosen entity or for the directive’s target) would let the model say "Last time you wrote to Marcus you said X; he replied Y; suggest a follow-up." That’s one new data surface (threaded messages per entity or per thread_id) with large impact on perceived intelligence for the core "draft email → approve → send" flow.

### 3.3 Single highest-leverage change (see §5)

Persist and use **extracted goals and outcomes** so the graph is complete and the model stops guessing on goals and past results.

## 4. Genius already in the code

*What’s sophisticated or novel.*

### 4.1 Approval-rate suppression with hard-suppress

**Location:** `lib/briefing/generator.ts` (buildApprovalRateSection)

Not only does the engine use feedback_weight (net per action_type), it separately computes **approval rate** per type and marks types as SUPPRESSED (<30% and ≥3 actions) and **HARD-SUPPRESSED** (3 consecutive skips). The prompt then tells the model not to generate those types unless evidence is overwhelming and qualitatively different. That’s a clear, interpretable policy that prevents the system from repeatedly suggesting something the user has consistently dismissed. Many products would only do "show less of this" without the "stop unless different" rule.

### 4.2 Cooling relationships as a first-class input

**Location:** `lib/relationships/tracker.ts`, `lib/briefing/generator.ts`

Relationship health is computed from **email volume over two windows** (last 30d vs prior 30d), with trend (increasing/stable/cooling), and only **cooling** contacts are passed into the conviction prompt. So the engine doesn’t get "all contacts"; it gets "people you’re drifting from." That’s a sharp product choice: the directive is biased toward re-engagement, not noise. The implementation (From/To parsing, counting, ratio, upsert into entity patterns) is straightforward but the **design**—surfacing only cooling—is what makes it leverage.

### 4.3 Artifact as the product, not the to-do

**Location:** `lib/conviction/artifact-generator.ts`, product docs

The product rule "The directive without an artifact is a to-do list. The artifact is the product" is implemented: every action type has a concrete output (email body, document text, calendar slot, research brief, decision frame, affirmation). Approval then **executes** that artifact (send email, create event, save doc). So the user never has to "do the work after saying yes"—Foldera does it. That’s the core differentiator and it’s wired through execution, not just UX copy.

### 4.4 Self-improving acquisition scoring

**Location:** `lib/acquisition/learning-loop.ts`, `lib/acquisition/scorer.ts`

Outreach scoring isn’t fixed: weights (keyword, ICP, intensity) are stored in `tkg_signals` (source=scoring_model, type=config) and updated by an analyzer that looks at approve/skip on outreach actions. So the system that surfaces "which post to reply to" gets better with use. That’s a closed loop for a different funnel (acquisition) than conviction, but the pattern—store config as signals, load in the hot path, update from behavior—is the same idea as the conviction feedback loop.

### 4.5 Skip reasons as structured feedback

**Location:** `lib/conviction/execute-action.ts` (SkipReason), generator CONVICTION_SYSTEM

Skip reason is typed (`not_relevant`, `already_handled`, `wrong_approach`) and the system prompt explains how to react to each. So the model isn’t just "they skipped"; it’s "they said wrong_approach → try a different action_type or angle." That turns a single bit (skip) into a small control panel for the next suggestion.

### 4.6 Single execution layer for conviction and DraftQueue

**Location:** `lib/conviction/execute-action.ts`

Both dashboard conviction (pending_approval) and DraftQueue (draft) go through `executeAction()`. One place handles approve/skip, DB updates, feedback signals, and artifact execution (email, calendar, document, etc.). That avoids drift between "how we execute from the dashboard" vs "how we execute from the queue" and keeps the learning loop consistent.

## 5. Output summary

### Top 3 intelligence upgrades (with implementation sketch)

1. **Persist and use extracted goals and outcomes**
   - In `conversation-extractor.ts` after parsing payload: for each `payload.goals`, insert into `tkg_goals` with source = `uploaded_document` or `email`, source_id = signal.id, optional priority from existing logic. For each `payload.outcomes`, either (a) find commitment by matching description (fuzzy or embedding later), update `resolution` and status, or (b) insert a signal type `outcome_confirmed` with decision_description + result so the generator can include "Past outcomes" in the context block.
   - In `generator.ts`, add an "OUTCOMES" or "CONFIRMED RESULTS" block when such data exists (from commitments.resolution or from signals).

2. **Ground confidence in evidence**
   - In `generator.ts`, before calling Claude: compute `approval_rate_by_type`, `net_weight_by_type`, and optionally a simple similarity score (e.g. keyword overlap of directive/reason with past approved directives). Compute `confidencePrior = f(approval_rate, net_weight)` (e.g. 50 + 20*approval_rate + 5*clip(net_weight, -2, 2)). Pass in the prompt: "Suggested confidence prior: {confidencePrior}. You may adjust ±15 based on qualitative evidence; do not exceed 95."
   - Optionally cap the model’s returned confidence at prior + 15 so it can’t overstate.

3. **Relationship-aware commitments in the prompt**
   - In `generator.ts`, extend the commitments query to join `tkg_entities` on promisor_id/promisee_id (or run a second query: commitments with entity names). Format commitment lines as "To/From [name]: description (risk, status)". Optionally add a short "Commitments by relationship" block so the model can tie "follow up with Marcus" to "you have an at_risk commitment to Marcus."

### Single highest-leverage change in the entire system

**Persist extracted goals and outcomes and feed them into the conviction prompt.**

Today the model is told it has "complete behavioral history" but two of the four extraction outputs (goals, outcomes) are thrown away. Goals drive what "matters"; outcomes drive what "worked." Restoring those two and surfacing them in the generator would raise the ceiling on every directive with no new data source—just using what’s already extracted.

### What this system does that is actually impressive

- **Closes the loop from behavior to suggestion:** Approval, skip, skip reason, and outcome (worked/didn’t work) all flow back into the next directive. Types and patterns that get rejected are suppressed; high-approval types are favored. That’s real learning, not a static recommender.
- **Treats the artifact as the product:** The system doesn’t stop at "you should email Marcus." It produces the actual email and, on approve, sends it. Same for calendar, documents, research. One-tap approval means the user doesn’t have to do the work—Foldera does. That’s the core product promise implemented in code.
- **Cooling relationships as a product feature:** Instead of "all contacts" or "recent contacts," the system computes trend and surfaces only people the user is drifting from. That’s a sharp, defensible choice that makes the engine useful instead of noisy.
- **Structured skip reasons that change behavior:** Not just "dismissed" but "not_relevant," "already_handled," "wrong_approach" with prompt rules so the next suggestion can pivot. That’s treating feedback as a control signal, not a single bit.
- **One execution and feedback path:** Conviction and DraftQueue share the same execute/feedback path, so learning and execution stay consistent and the codebase stays maintainable.
